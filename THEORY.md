# Optimo-Algo: Theoretical Architecture

## 1. Executive Summary
**Optimo-Algo** is a bridging layer and token-optimization tool designed to drastically reduce API costs and context window usage for high-tier AI coding agents (e.g., GPT-4, Claude 3.5 Sonnet, Gemini 1.5 Pro). By utilizing a low-parameter local model running via **Ollama**, it pre-processes a massive codebase, extracting only the critical structural and logical context. The distilled context is then packaged into a proprietary format called **TOON (Token Oriented Object Notation)** and passed to the heavy-lifting cloud agent.

## 2. Core Components

### 2.1 The Local Engine (Ollama)
Instead of sending raw code with thousands of lines of boilerplate, whitespace, and irrelevant comments to expensive APIs, Optimo-Algo uses an ultra-fast local LLM (e.g., Llama-3-8B or Qwen-2.5-7B) running via Ollama. 
- **Task:** The local model reads the codebase file by file and generates structural summaries. It identifies classes, function signatures, input/output types, and the core purpose of the file without retaining the line-by-line implementation.
- **Advantage:** Compute is traded for token savings. Your local machine does the heavy scanning for free, while the cloud API is only invoked for high-level reasoning and final implementation.

### 2.2 TOON (Token Oriented Object Notation)
The resulting context is compiled into a `.toon` file. TOON is a highly minified, token-dense representation of a codebase specifically designed to be read by LLMs, not humans. 
- **Format:** It acts as a hyper-compressed JSON/YAML hybrid, optimized for LLM tokenizers (avoiding whitespace, using common token strings, and abbreviating keys).
- **Structure:**
  - `d`: Directory
  - `f`: File
  - `c`: Class
  - `m`: Method/Function (includes args and return type)
  - `s`: Summary (A 1-2 sentence distillation of what the component does, generated by the local model).

**Example of a `.toon` structure:**
```json
{"d":"src","c":[{"f":"auth.py","c":[{"n":"AuthHandler","m":[{"n":"login","a":["user","pass"],"r":"bool","s":"Authenticates UI via JWT"}]}]}]}
```
*Note how a 500-line authentication file is reduced to ~30 tokens of pure context.*

## 3. The Optimo-Algo Workflow

1. **Ingestion:** The user defines the working directory. Optimo-Algo maps the standard folder tree (ignoring `.git`, `node_modules`, etc.).
2. **Local Distillation:** Optimo-Algo queries the local Ollama instance sequentially or in parallel batches. The prompt to Ollama is strict: *"Summarize the structural logic of this code. Output strictly as TOON objects."*
3. **Compilation:** The individual outputs are stitched together into an overarching `context.toon` file.
4. **Handoff:** The user inputs their desired feature request (e.g., "Add a password reset flow"). Optimo-Algo sends the user prompt PLUS the `context.toon` file to the Cloud Agent.
5. **Execution:** The Cloud Agent, now armed with perfect structural awareness of the entire project but without the token bloat of the raw source code, can accurately pinpoint which files need to be modified. It writes the exact necessary code, which is then applied to the local repository.

## 4. Why This Works
Modern frontier LLMs are incredibly good at inferring implementation details if they understand the **architecture** and the **signatures**. They do not need to read 10,000 lines of standard React components to know how to add a new route. By feeding them a `.toon` map, you satisfy their contextual requirement while saving up to 90% on input tokens.
